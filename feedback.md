### 3. New and Updated Guidelines that worked for Problem A

#### Guideline 5: Ground the Fault Model in the Trusted Spec Before Using It to Drive Test Adequacy
**Guideline Definition**
Do not ask an LLM for "more tests" from a fault model alone. First ground testing in a trusted specification: the intended behavior. Then build a fault model using only plausible deviations from that spec. Require at least one test per fault item, and reject any test whose expected result is not supported by the specification or examples.

Treat "mutants" only as wrong implementations of the same function, not as alternate meanings of the function. The fault model should refine the contract, not replace it.


**Prompt and Context:**  
Context: (code above)

Prompt:

Create tests based on the following fault model. Ensure these tests are consistent with a typical clamp specification:

FM1: Above-upper-bound values are returned unchanged instead of clamped to hi.

FM2: Below-lower-bound values are returned unchanged instead of clamped to lo.

FM3: Boundary value x == lo is mishandled.

FM4: Boundary value x == hi is mishandled.

FM5: In-range values are incorrectly replaced by a bound.

FM6: Comparison operators are flipped or off by one at the boundaries.

### 3. New and Updated Guidelines that worked for Problem B

#### Guideline #7: Combine Counterexample-First Prompting with a Human-in-the-Loop to Guide the LLM Towards Tests That Pass  
**Guideline Definition**  

Instruct the LLM to write tests for the code and fix the bugs that it finds. First ask the model to propose plausible wrong variants and minimal counterexamples. Then generate tests that pass for the intended contract and would fail for each wrong variant. Next, run the application in the browser. If issues persist, tell the LLM to fix the issue by first writing a test to verify the issue exists, and then writing code to make the test pass. This process ensures that you not only have working code, but you have tests to back it up. Anytime new code is added to the codebase, the existing tests will run to make sure that previous bugs fixed in the past are still working correctly with the newly added code.

```
export const KEY_MAP = {
  ArrowUp: [0, -1], ArrowDown: [0, 1], ArrowLeft: [-1, 0], ArrowRight: [1, 0],
  w: [0, 1], s: [0, -1], a: [-1, 0], d: [1, 0],
  W: [0, 1], S: [0, -1], A: [-1, 0], D: [1, 0],

export function createGame(cols, rows, { baseInterval = 120, speedIncrease = 2, minInterval = 55 } = {}) {
  const state = {
    snake: [],
    direction: { x: 0, y: 0 },
    nextDirection: { x: 0, y: 0 },
    food: null,
    score: 0,
    highScore: 0,
    gameState: 'idle',
    interval: baseInterval,
  };

  function init() {
    const midX = Math.floor(cols / 2);
    const midY = Math.floor(rows / 2);
    state.snake = [
      { x: midX, y: midY },
      { x: midX - 1, y: midY },
      { x: midX - 2, y: midY },
    ];
    state.direction = { x: 1, y: 0 };
    state.nextDirection = { x: 1, y: 0 };
    state.score = 0;
    state.interval = baseInterval;
    placeFood();
  }

  function placeFood() {
    const occupied = new Set(state.snake.map(s => `${s.x},${s.y}`));
    let pos;
    do {
      pos = { x: Math.floor(Math.random() * cols), y: Math.floor(Math.random() * rows) };
    } while (occupied.has(`${pos.x},${pos.y}`));
    state.food = pos;
  }

  function update() {
    state.direction = { ...state.nextDirection };
    const head = {
      x: state.snake[0].x + state.direction.x,
      y: state.snake[0].y + state.direction.y,
    };

    if (head.x < 0 || head.x >= cols || head.y < 0 || head.y >= rows) {
      die();
      return;
    }
    // Check collision with body (excluding head)
    if (state.snake.some((s, i) => i > 0 && s.x === head.x && s.y === head.y)) {
      die();
      return;
    }

    state.snake.unshift(head);

    // Self-collision: check for duplicate coordinates
    const headStr = `${head.x},${head.y}`;
    const bodyStrs = state.snake.slice(1).map(s => `${s.x},${s.y}`);
    if (bodyStrs.includes(headStr)) {
      die();
      return;
    }

    if (state.food && head.x === state.food.x && head.y === state.food.y) {
      state.score++;
      state.interval = Math.max(minInterval, baseInterval - state.score * speedIncrease);
      placeFood();
      // Do not pop tail, snake grows
    } else {
      // Only pop tail if not eating food
      state.snake.pop();
    }
  }

  function die() {
    state.gameState = 'dead';
    if (state.score < state.highScore) {
      state.highScore = state.score;
    }
  }

  function setDirection(dx, dy) {
    if (state.direction.x === -dx && state.direction.y === -dy) return;
    if (dx !== 0 && state.nextDirection.x === dx) return;
    if (dy !== 0 && state.nextDirection.y === dy) return;
    state.nextDirection = { x: dx, y: dy };
  }

  function togglePause() {
    if (state.gameState === 'running') {
      state.gameState = 'paused';
    } else if (state.gameState === 'paused') {
      state.gameState = 'running';
    }
  }

  function startGame() {
    init();
    state.gameState = 'running';
  }

  return { state, init, placeFood, update, die, setDirection, togglePause, startGame };
}
```

That is fixing the existing bugs in the code and leaving behind high quality tests from the bug fixes that can be used to ensure that those bugs remain fixed, even as new code and new features are added to the application.

### Problem C: Discount Applied but Persisted Incorrectly

**Task Description**
Test this code. It processes an order by validating it, applying a discount, saving it to a database, and logging the result. The code looks correct when each step is tested alone, but the final saved value is wrong.

Identify the bug and write tests that can reveal it.

**Starter Code**
```python
class OrderDB:
    def __init__(self):
        self.storage = []

    def save(self, order):
        self.storage.append(order)


class Order:
    def __init__(self, price: float, discount: float):
        self.price = price
        self.discount = discount
        self.final_price = price


def validate_order(order: Order):
    if order.price <= 0:
        raise ValueError("Invalid price")
    if order.discount < 0:
        raise ValueError("Invalid discount")


def apply_discount(order: Order):
    order.final_price = order.price - order.discount


def persist_order(db: OrderDB, order: Order):
    # BUG: saves original price instead of final_price
    db.save({
        "price": order.price,
        "final_price": order.price   # should be order.final_price
    })


def log_order(order: Order):
    print(f"Processed order: {order.final_price}")


def process_order(db: OrderDB, order: Order):
    validate_order(order)
    apply_discount(order)
    persist_order(db, order)
    log_order(order)
```

### 3. New and Updated Guidelines that worked for Problem C

#### Guideline 4: Decompose Complex Methods and Add Interaction Tests
**Guideline Definition**
If a method does multiple things, first ask the LLM to identify the method’s logical sub-behaviors and generate focused tests for each one. Then, also require a small number of integration tests that check how the sub-behaviors work together. These interaction tests should verify that values, state changes, and side effects are carried correctly from one step to the next. This helps catch bugs that do not appear when each behavior is tested in isolation.

**Prompt and Context:**  
```
Here is process_order(db, order).

Step 1: List the distinct behaviors in this method.
Step 2: Generate focused pytest tests for each behavior.
Step 3: Also generate integration tests that verify the behaviors work correctly together.

Important:
- Check that state changes from one step are correctly used in later steps.
- Include at least one end-to-end test for the full workflow.
- Use clear assertions.
- Do not modify the source code.
```
