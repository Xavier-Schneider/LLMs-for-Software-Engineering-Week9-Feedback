### 3. New and Updated Guidelines that worked for Problem A

#### Guideline 5: Ground the Fault Model in the Trusted Spec Before Using It to Drive Test Adequacy
**Guideline Definition**
Do not ask an LLM for "more tests" from a fault model alone. First ground testing in a trusted specification: the intended behavior. Then build a fault model using only plausible deviations from that spec. Require at least one test per fault item, and reject any test whose expected result is not supported by the specification or examples.

Treat "mutants" only as wrong implementations of the same function, not as alternate meanings of the function. The fault model should refine the contract, not replace it.


**Prompt and Context:**  
Context: (code above)

Prompt:

Create tests based on the following fault model. Ensure these tests are consistent with a typical clamp specification:

FM1: Above-upper-bound values are returned unchanged instead of clamped to hi.

FM2: Below-lower-bound values are returned unchanged instead of clamped to lo.

FM3: Boundary value x == lo is mishandled.

FM4: Boundary value x == hi is mishandled.

FM5: In-range values are incorrectly replaced by a bound.

FM6: Comparison operators are flipped or off by one at the boundaries.

**Solution or Output generated by GitHub Copilot:**  
```
# targets: FM1 above-upper-bound returned unchanged
# spec support: "returns hi if x > hi"; trusted example clamp(99, 0, 10) == 10
def test_clamp_above_upper_bound():
    assert clamp(11, 0, 10) == 10


# targets: FM2 below-lower-bound returned unchanged
# spec support: "returns lo if x < lo"; trusted example clamp(-1, 0, 10) == 0
def test_clamp_below_lower_bound():
    assert clamp(-1, 0, 10) == 0


# targets: FM3 lower-bound equality mishandled
# spec support: "returns x otherwise"; trusted example clamp(0, 0, 10) == 0
def test_clamp_at_lower_bound():
    assert clamp(0, 0, 10) == 0


# targets: FM4 upper-bound equality mishandled
# spec support: "returns x otherwise"; trusted example clamp(10, 0, 10) == 10
def test_clamp_at_upper_bound():
    assert clamp(10, 0, 10) == 10


# targets: FM5 in-range value replaced incorrectly
# spec support: "returns x otherwise"; trusted example clamp(5, 0, 10) == 5
def test_clamp_in_range_value():
    assert clamp(5, 0, 10) == 5


# targets: FM6 boundary comparison off-by-one / wrong operator
# spec support: values strictly greater than hi clamp to hi
def test_clamp_just_above_upper_bound():
    assert clamp(1, 0, 0) == 0
```

*** Analysis ***

In the fixed version, the spec is defined prior to the fault model, the fault model is subordinate to the spec and the tests must justify expected outputs from the spec, not from the fault model alone. This prevents the bad failure mode where the LLM confidently tests for wraparound, reflection, or some other invented behavior while missing the real bug.

### Problem C: Discount Applied but Persisted Incorrectly

**Task Description**
Test this code. It processes an order by validating it, applying a discount, saving it to a database, and logging the result. The code looks correct when each step is tested alone, but the final saved value is wrong.

Identify the bug and write tests that can reveal it.

**Starter Code**
```python
class OrderDB:
    def __init__(self):
        self.storage = []

    def save(self, order):
        self.storage.append(order)


class Order:
    def __init__(self, price: float, discount: float):
        self.price = price
        self.discount = discount
        self.final_price = price


def validate_order(order: Order):
    if order.price <= 0:
        raise ValueError("Invalid price")
    if order.discount < 0:
        raise ValueError("Invalid discount")


def apply_discount(order: Order):
    order.final_price = order.price - order.discount


def persist_order(db: OrderDB, order: Order):
    # BUG: saves original price instead of final_price
    db.save({
        "price": order.price,
        "final_price": order.price   # should be order.final_price
    })


def log_order(order: Order):
    print(f"Processed order: {order.final_price}")


def process_order(db: OrderDB, order: Order):
    validate_order(order)
    apply_discount(order)
    persist_order(db, order)
    log_order(order)
```

### 2. Guidelines that didn't work for Problem C

#### Guideline 4: Decompose Complex Methods Before Asking for Tests

**Guideline Definition**
If a method does multiple things (e.g., validate → transform → persist → log), don’t ask the LLM to “write tests” in one shot. Instead:
* Ask the LLM to identify the method’s logical sub-behaviors (a short list).
* Generate separate focused tests per sub-behavior (one responsibility per test group).
* Keep categories explicit inside each sub-behavior.
This keeps the model from mixing concerns and makes it easier to audit coverage.

**Prompt and Context:**  
```
Here is process_order(db, order).

Step 1: List the distinct behaviors in this method.
Step 2: For each behavior, generate pytest tests focusing only on that behavior.
Step 3: For each behavior, include boundary + negative cases.

Constraints:
- clear assertions
- do not test unrelated code
```

### 3. New and Updated Guidelines that worked for Problem C

#### Guideline 4: Decompose Complex Methods and Add Interaction Tests
**Guideline Definition**
If a method does multiple things, first ask the LLM to identify the method’s logical sub-behaviors and generate focused tests for each one. Then, also require a small number of integration tests that check how the sub-behaviors work together. These interaction tests should verify that values, state changes, and side effects are carried correctly from one step to the next. This helps catch bugs that do not appear when each behavior is tested in isolation.

**Prompt and Context:**  
```
Here is process_order(db, order).

Step 1: List the distinct behaviors in this method.
Step 2: Generate focused pytest tests for each behavior.
Step 3: Also generate integration tests that verify the behaviors work correctly together.

Important:
- Check that state changes from one step are correctly used in later steps.
- Include at least one end-to-end test for the full workflow.
- Use clear assertions.
- Do not modify the source code.
```

**Solution or Output generated by GitHub Copilot:**  

```python
import os
import sys
import pytest

ROOT = os.path.abspath(os.path.join(os.path.dirname(__file__), ".."))
if ROOT not in sys.path:
    sys.path.insert(0, ROOT)

from ProblemC import process_order, OrderDB, Order


def test_process_order_integration_persists_and_logs(capsys):
    db = OrderDB()
    o = Order(200.0, 20.0)
    process_order(db, o)

    # State change: final_price should be updated by apply_discount
    assert o.final_price == 180.0

    # Persisted record should contain the final_price (end-to-end expectation)
    assert len(db.storage) == 1
    saved = db.storage[0]
    assert saved["price"] == 200.0
    assert saved["final_price"] == 180.0

    # Logging should reflect the final price
    captured = capsys.readouterr()
    assert f"Processed order: {o.final_price}" in captured.out


def test_multiple_orders_end_to_end():
    db = OrderDB()
    o1 = Order(30.0, 5.0)
    o2 = Order(100.0, 25.0)
    process_order(db, o1)
    process_order(db, o2)

    assert len(db.storage) == 2
    assert db.storage[0]["final_price"] == 25.0
    assert db.storage[1]["final_price"] == 75.0
```

**Analysis:**

By testing how the steps interact, the model was able to detect the problem that isolated unit tests alone could miss. This shows that combining focused unit tests with integration tests leads to better bug detection for multi-step methods.
